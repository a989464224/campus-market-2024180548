## 1. 今日完成内容

今天完成了"校园轻集市"发布功能的前端实现，采用类型切换表单方案，支持四类业务信息的发布。具体包括：

- 在 4 个 API 模块中新增 `create` 方法（`createTrade`、`createLostFound`、`createGroupBuy`、`createErrand`），统一使用 POST 请求写入 JSON Server；
- 创建可复用的 `FormField.vue` 表单控件组件，统一表单项的标签、必填标识和错误提示；
- 重写 `PublishView.vue`，用类型切换表单替代原有的三步向导，根据用户选择的发布类型动态展示对应的关键字段；
- 实现基础表单校验逻辑，覆盖必填项检查和数值范围校验；
- 实现提交逻辑，调用对应 API 写入数据，成功后跳转到对应列表页；
- 修复了发布后路由跳转失败的问题（路由路径不匹配）和我的发布页面数据不更新的问题（数据硬编码）。

## 2. 发布功能设计说明

### 2.1 方案选择

| 实现方式 | 说明 | 适合情况 |
|---|---|---|
| 简化统一表单 | 所有类型共用标题、描述、地点等通用字段 | 基础较弱 |
| **类型切换表单** ✅ | 根据发布类型显示不同字段 | 基础较好 |

本实训采用**类型切换表单**：通过 `v-if="publishType === 'trade'"` 等条件渲染，根据用户在下拉框中选择的发布类型，动态展示对应的关键字段。不引入复杂动态表单引擎，保持代码可读性。

### 2.2 四类业务的字段设计

| 发布类型 | 对应数据集合 | 关键字段 | 字段设计理由 |
|---|---|---|---|
| 二手交易 | trades | 标题、分类、价格、成色、地点、描述 | 二手交易核心是**价格**和**成色**——买家需要快速判断是否在预算内、商品折旧程度；分类帮助筛选 |
| 失物招领 | lostFounds | 标题、类型（寻物/招领）、物品名称、地点、时间、描述 | 寻物和招领是两种方向的信息，用一个 `type` 字段区分避免数据模型碎片化；物品名称和发生时间是匹配失主的关键 |
| 拼单搭子 | groupBuys | 标题、类型、目标人数、截止时间、地点、描述 | 拼单的核心逻辑是"人够成团"，targetCount 和 deadline 是两个必要约束；currentCount 提交时初始化为 1 |
| 跑腿委托 | errands | 标题、任务类型、酬劳、取件地点、送达地点、截止时间、描述 | from/to 描述任务起止点，帮助接单人判断是否顺路；reward 是接单的核心激励 |

### 2.3 通用字段说明

四类业务共用 `title`、`location`、`description` 三个通用字段。其中 `publisher`（发布人）在 Day4 阶段固定为 `"当前用户"`，等待 Day5 结合 Pinia 模拟真实用户状态。`id` 由 JSON Server 自动生成，不在表单中填写。

## 3. 表单校验规则设计

| 校验项 | 适用类型 | 规则 | 理由 |
|---|---|---|---|
| 标题 | 全部 | 必填 | 无标题的信息无法在列表中识别 |
| 地点 | 全部 | 必填 | 校园交易依赖于地理位置（面交、取件地点） |
| 描述 | 全部 | 必填 | 提供必要的上下文信息，避免只有标题的"空壳"信息 |
| 商品分类 | trade | 必填 | 帮助浏览者按分类筛选 |
| 价格 | trade | 必须 > 0 | 二手商品应有明确标价，0 元或负数无意义 |
| 成色 | trade | 必选 | 成色是二手交易区别于新货的核心指标 |
| 物品名称 | lostFound | 必填 | 失主/拾主需要明确知道是什么物品 |
| 发生时间 | lostFound | 必填 | 丢失/拾获时间是判断信息时效性的关键 |
| 拼单类型 | groupBuy | 必填 | 区分拼餐、拼车、团购等不同场景 |
| 目标人数 | groupBuy | 必须 ≥ 2 | 至少 2 人才能称为"拼" |
| 截止时间 | groupBuy / errand | 必填 | 超时的拼单/委托不应继续展示 |
| 任务类型 | errand | 必填 | 区分取快递、代买、代送等不同任务 |
| 酬劳 | errand | 必须 ≥ 0 | 酬劳不能为负数（允许 0 元互助委托） |
| 取件地点 | errand | 必填 | 接单人需要知道从哪里取 |
| 送达地点 | errand | 必填 | 接单人需要知道送到哪里 |

## 4. 代码审查（7 项审查清单）

### 4.1 字段是否贴合业务 ✅

经过审查，四类业务的字段均贴合校园场景：
- 二手交易有 price 和 condition（成色），符合二手商品特征；
- 失物招领有 type（lost/found）和 itemName，满足寻物和招领两种需求；
- 拼单搭子有 targetCount 和 deadline，支撑"人够成团"的核心逻辑；
- 跑腿委托有 from/to 描述起止地点，reward 描述酬劳。

### 4.2 校验是否必要 ✅

所有校验规则（详见第 3 节）均在 `validateForm()` 函数中实现，每条规则都有明确的业务理由。校验失败时通过 `errors` 响应式对象将错误信息传递到 `FormField` 组件的 `error` 属性中，在对应字段下方显示红色提示文字。

### 4.3 提交路径是否正确 ✅

| 发布类型 | API 方法 | POST 路径 | 对应 JSON Server 集合 |
|---|---|---|---|
| 二手交易 | createTrade() | POST /trades | trades |
| 失物招领 | createLostFound() | POST /lostFounds | lostFounds |
| 拼单搭子 | createGroupBuy() | POST /groupBuys | groupBuys |
| 跑腿委托 | createErrand() | POST /errands | errands |

所有提交路径均与 db.json 中的数据集合一一对应，JSON Server 自动处理路由映射。

### 4.4 数据字段是否一致 ✅

提交数据字段与 db.json 中已有字段完全一致。提交时：
- 使用 TypeScript 接口（TradeItem、LostFoundItem 等）约束参数类型；
- 发布人固定为 `"当前用户"`；
- 发布时间使用 `getCurrentTime()` 生成 `YYYY-MM-DD HH:mm` 格式；
- 状态初始化为 `"open"`；
- 拼单的 currentCount 初始化为 1（发布者自身）。

### 4.5 交互反馈是否清晰 ✅

- **提交中**：按钮文字变为"提交中..."并禁用，防止重复提交；
- **成功**：`window.alert()` 弹出成功提示（如"二手商品发布成功"），随后自动跳转；
- **失败**：catch 块中弹出"发布失败，请检查 Mock 服务是否正常运行"；
- **校验失败**：对应字段下方显示红色错误文字。

### 4.6 跳转是否合理 ✅

| 发布类型 | 跳转目标 | 路由路径 |
|---|---|---|
| 二手交易 | 二手交易列表页 | /trade（补充注册） |
| 失物招领 | 失物招领列表页 | /lost-found |
| 拼单搭子 | 拼单搭子列表页 | /group-buy |
| 跑腿委托 | 跑腿委托列表页 | /errand |

审查中发现 AI 初始生成的代码使用 `/trade` 路径，但路由表中原有的二手交易页面注册在 `/category`，导致跳转失败。**人工修复**：在路由表中新增 `/trade` 路由指向 TradeView.vue，确保跳转正常。

### 4.7 代码是否简洁 ✅

- 未引入复杂表单校验库（如 VeeValidate），仅使用手写 `validateForm()` 函数；
- 未处理真实用户鉴权，发布人固定为 `"当前用户"`；
- 未处理图片上传，image 字段固定为空字符串；
- 未涉及真实后端逻辑，全部通过 JSON Server Mock；
- `FormField.vue` 组件仅 45 行，职责单一（标签 + 插槽 + 错误提示）；
- `PublishView.vue` 使用条件渲染（`v-if`）而非动态表单引擎，逻辑清晰可读。

## 7. 遇到的问题与解决方法

### 问题一：发布后页面不跳转到交易页面

**现象**：点击"发布"按钮后，alert 提示"二手商品发布成功"，但页面停留在发布页不跳转。

**排查过程**：
1. 打开浏览器开发者工具 Network 面板，确认 POST 请求成功（HTTP 201）；
2. 检查 `router.push('/trade')` 是否执行——在代码中插入 `console.log` 确认该行已执行；
3. 检查路由配置——发现 `/trade` 路径在路由表中不存在，只有 `/category` 指向 TradeView.vue。

**解决方法**：在 `src/router/index.ts` 中新增 `/trade` 路由，同时保留原有 `/category` 路由，两者均指向 TradeView.vue，确保跳转正常。

### 问题二：个人中心"我的发布"数量显示 12，点进去只有 7 条

**现象**：ProfileView 显示"我的发布 12"，但点进 UserView 后实际只看到 7 条二手交易数据。

**排查过程**：
1. 检查 ProfileView.vue 代码——第 4 行 `stats = ref({ posts: 12, ... })` 硬编码为 12；
2. 检查 UserView.vue 代码——发现虽然已重写为从 API 获取数据，但只获取了 trades 而未获取其他三类；
3. 早期版本的 UserView 使用的是写死的 5 条假数据。

**解决方法**：重写 UserView.vue 从 4 个 API 并行获取全部数据，合并为统一列表；同时修改 ProfileView.vue 从 API 动态获取真实发布总数。

### 问题三：AI 生成的 PublishView 是三步向导，不是类型切换表单

**现象**：AI 第一版生成的 PublishView.vue 使用了 `el-steps` 三步向导（选择类型→填写信息→确认发布），而实训要求的是类型切换表单。

**解决方法**：重新向 AI 提供实训文档中的代码片段和设计说明，生成符合要求的类型切换表单版本（一个下拉框切换类型 + 条件渲染不同字段组）。

## 8. 今日反思

发布功能是"校园轻集市"项目从"只读浏览"迈向"数据写入"的关键转折点。今天的实训让我深刻理解了几个工程要点：

**类型切换表单 vs 统一表单的取舍**：四类业务的字段确实存在差异——二手交易需要价格和成色，失物招领需要物品名称和发生时间，拼单需要目标人数和截止时间，跑腿需要起止地点和酬劳。如果强行使用统一表单，要么字段过多导致表单臃肿，要么丢失关键字段导致信息不完整。类型切换方案在保持代码简洁的前提下，为每类业务提供了精确的字段支持。

**路由设计的严谨性**：路由路径的前后一致看起来是小事，但一旦出现不匹配，用户操作就会静默失败——没有报错，页面就是不跳转。这种"静默失败"是最难排查的 bug，也让我意识到命名规范（如 trade vs category）在团队协作中的重要性。

**数据流的完整性**：Day3 我们实现了"从 API 读数据"，Day4 实现了"向 API 写数据"，但写完数据后还要确保"读数据的页面能反映最新状态"。UserView 和 ProfileView 的硬编码问题说明，只实现写入逻辑是不够的，必须端到端验证：发布→写入 db.json→列表页重新请求→看到新数据。缺少任何一个环节，用户感知到的就是"功能坏了"。

**AI 协作的边界**：AI 能高效生成大段代码（校验函数、表单模板、API 方法），但无法自主发现路由不匹配、数据硬编码这类"跨文件的逻辑不一致"问题。这些问题需要人工在审查时逐项对照——对照路由表和跳转路径、对照 API 返回值和页面响应式变量——才能发现。这也印证了实训的核心理念：**"AI 提效，人工把关"**。
