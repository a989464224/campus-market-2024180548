1. 为什么要开发“校园轻集市”？
开发“校园轻集市”的核心目的，不是交付一个可以上线运营的真实产品，而是搭建一个服务于教学的前端工程实践载体。
具体有三个层面的原因：
· 提供真实的教学场景：大学校园里的二手交易、失物招领等是学生熟悉的高频场景。用这些作为项目背景，学生能快速理解业务，把精力集中在技术实现上，而不用花时间去弄懂陌生业务。
· 串联现代前端技术栈：项目被设计成一个“锚点”，用以串联 Vue3、Vite、Pinia、Axios、Element Plus 等现代前端工具。学生不是为了学技术而学，而是在“做中学”，在解决具体功能需求时，自然而然地掌握这些技术的工程价值。
· 培养 AI 时代的工程素养：这是本项目最独特的定位。它不仅训练编码，更核心的是训练学生与 AI 高效协作完成需求、代码、测试等任务，同时培养代码审查、问题定位和使用 Git 进行版本回退的工程习惯。

2. 系统主要解决哪些实际问题？
系统分为两个层面，一是项目内部模拟的校园生活问题，二是项目对学生自身能力训练解决的问题。
· 模拟解决的校园生活问题（项目内部场景）：
  · 二手交易：解决校内闲置物品（教材、电子产品等）的流转问题。
  · 失物招领：解决物品遗失与拾获的信息匹配问题。
  · 拼单搭子：解决奶茶、外卖等临时性协作伙伴的寻找问题。
  · 跑腿委托：解决代取快递、代送文件等临时任务的需求对接问题。
· 重点解决的学生能力问题（实训的真正目的）：
  · 从静态页面到工程化开发的跨越：帮助学生理解前端如何通过 API 与后端交互，告别“数据写死在代码里”的初级阶段。
  · “AI 提效，人工把关”的协作能力：训练学生与 AI 高效协作，能向 AI 描述需求、审查并修改其生成的代码。
  · 基础的工程风险控制能力：通过 Git，培养学生“先提交，再修改，出问题可回退”的工程安全习惯。

3. 系统包含哪些业务场景？
项目围绕四大业务场景展开，但出于教学目的，它们被巧妙地统一在一个数据模型下。它们分别是：
· 二手交易：商品型信息。训练重点是价格、成色、收藏、模拟砍价和交易状态更新。
· 失物招领：信息型互助。训练重点是物品特征、丢失地点、时间和找回状态的管理。
· 拼单搭子：多人参与型信息。训练重点是目标人数、截止时间和参与状态的展示。
· 跑腿委托：任务型信息。训练重点是任务内容、地点、酬劳和完成状态的流转。
这四类场景在代码层面被统一抽象为 items 资源，通过 type 字段来区分。这种“统一建模，差异化展示”的设计，既保证了教学案例的丰富性，又有效控制了项目的复杂度。

4. 七天实训最终要完成什么？
七天的实训，学生最终要交付一套完整、可演示的“校园轻集市”前端项目，并在此过程中完成能力蜕变。具体可以分为有形产出、能力目标和核心理念三个层面：
· 有形的项目产出：
  1. 一份可运行的源码：一个结构清晰的 Vue3 前端单页应用。
  2. 一份 Mock 数据文件：设计完整的 db.json 文件。
  3. 一份运行说明与演示材料：能让别人顺利运行项目，并辅以答辩PPT。
  4. 一份带有清晰提交历史的 Git 仓库：记录了从初始化到最终提交的完整开发过程。
· 无形的能力蜕变：
  1. 需求转化能力：能将“我想买二手书”这类生活需求，转化为用户故事和功能点。
  2. 工程化开发能力：能使用路由、状态管理、接口封装等方式构建清晰的应用。
  3. 前后端分离思想：深刻理解前端、API、Mock 服务三者间的职责和数据流转。
  4. 人机协同开发能力：能驾驭 AI 工具进行需求分析、编码、调试和测试。
  5. 项目展示与总结能力：能清晰地展示项目、阐述架构并复盘过程。
· 核心理念的升级：
  最终的成果，是让学生建立“功能服务教学、AI 服务开发、工具服务工程”的基本意识。他们完成的不仅是一个项目，更是一次现代前端工程师工作流的全真模拟，为后续深入学习全栈技术铺平了道路。
